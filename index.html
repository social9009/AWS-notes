DNS & How the Internet Works

🧠
Interview Gold: "What happens when you type www.google.com?" — This is asked in 80% of DevOps/Cloud interviews. Master all 10 steps!

📡 End-to-End Request Flow

  1. Browser Cache Check — Browser first checks local cache. Recently visited? Return instantly. Not cached? Proceed to DNS.

  2. OS Cache (/etc/hosts) — Checks OS-level DNS cache and /etc/hosts file. If found, use that IP immediately.

  3. DNS Resolution — Recursive resolver (ISP) → Root DNS → TLD (.com) → Authoritative DNS → Returns IP like 142.250.183.206

  4. TCP 3-Way Handshake — Client: SYN → Server: SYN-ACK → Client: ACK. Connection established!

  5. TLS Handshake (HTTPS) — Browser + Server agree on encryption algo, exchange certificates, verify identity. Prevents MITM attacks.

  6. HTTP Request Sent — GET / HTTP/1.1 Host: www.google.com with User-Agent, cookies, headers.

  7. Load Balancer (AWS ALB) — Request hits ALB first. ALB checks target group health checks, routes to healthy EC2/ECS instance.

  8. Server Processing — App processes request. May query RDS, hit ElastiCache, call DynamoDB, or invoke microservices.

  9. HTTP Response — Server sends HTTP/1.1 200 OK with HTML/CSS/JS payload.

  10. Browser Renders — Parses HTML → builds DOM → loads CSS → executes JS → 🎉 You see the website!

AWS NAT Gateway

🏢 Real Org Scenario: Backend EC2 in private subnet needs to pull npm packages from GitHub. Private subnet has NO internet access. Solution: NAT Gateway in public subnet. Private EC2 → NAT → IGW → Internet. Source IP never exposed. Completely secure outbound access.

🔄 SNAT Packet Flow — Step by Step

# Setup: Private EC2 (10.0.2.10) downloads from internet
# NAT EIP: 3.110.25.100

Step 1 — Private EC2 sends packet:
  SRC: 10.0.2.10:50000  →  DST: 8.8.8.8:443

Step 2 — Route Table decision:
  Private route: 0.0.0.0/0 → NAT Gateway  → packet forwarded

Step 3 — SNAT Translation:
  BEFORE: SRC = 10.0.2.10:50000  (private, hidden)
  AFTER:  SRC = 3.110.25.100:60000 (Elastic IP, public)
  NAT State Table: 3.110.25.100:60000 ↔ 10.0.2.10:50000

Step 4 — NAT → IGW → Internet → Reaches 8.8.8.8:443

Step 5 — Response comes back: DST = 3.110.25.100:60000
  NAT looks up state table → Reverse NAT
  DST: 3.110.25.100:60000 → 10.0.2.10:50000

Result:
  ✅ Private EC2 downloaded packages
  ✅ Private IP never exposed to internet
  ❌ Internet CANNOT initiate connection back to private EC2

Inbound internet✅ Allowed❌ Blocked
Outbound internet✅ Allowed✅ Allowed
Used byPublic subnetsPrivate subnets
Placed inVPC attachmentPublic subnet
TypeBidirectionalOutbound only (SNAT)

🚨
Most Common Mistake — NACL Stateless!
You MUST allow BOTH inbound (request) AND outbound (ephemeral ports 1024–65535) in NACL.
Forgetting ephemeral ports = connection timeout. Users can send but NEVER receive responses!

🎯 SG Referencing — Dynamic & Scalable

❌ IP-Based Rules (Wrong)

DB-SG inbound:
Allow 3306 from 10.0.1.10
Allow 3306 from 10.0.1.11  ← breaks on restart
Allow 3306 from 10.0.1.12  ← fails with Auto Scaling

Auto Scaling adds 15 new instances → ALL fail to connect. You must manually add 15 more IP rules.

✅ SG Referencing (Right)

DB-SG inbound:
Allow 3306 from APP-SG
# That's it! All instances with
# APP-SG are automatically allowed.
# Scale from 2 → 20 → zero changes!

AWS checks: "Does incoming ENI have APP-SG attached?" Not the IP. This is dynamic!

🔌 ENI Architecture

# ENI = Virtual Network Card attached to EC2
# IMPORTANT: Security Group attaches to ENI, NOT directly to EC2!

EC2 Instance
    └── ENI (Elastic Network Interface)
            ├── Private IP: 10.0.1.15
            ├── MAC Address: 0a:1b:2c:3d:4e:5f
            └── Security Groups: [WEB-SG, MONITORING-SG]  ← here!

# Advanced: Multi-ENI EC2 (Network Appliances)
EC2 Instance
    ├── ENI-1 → Public Subnet → [WEB-SG]      (internet traffic)
    └── ENI-2 → Private Subnet → [INTERNAL-SG] (backend traffic)

AWS RDS Deep Dive

🔄 Multi-AZ vs Read Replicas

🛡️ Multi-AZ — High Availability
PRIMARY DB · AZ-1a · (Active)
⚡ Synchronous → every write mirrors instantly
STANDBY DB · AZ-1b · (Passive — no reads!)
Replication: SYNCHRONOUS
Failover: AUTOMATIC (~60s)
Standby serves reads: NO
Data loss: Near zero
VS
📖 Read Replicas — Performance
PRIMARY DB (Writes)
async replication (lag possible)
Replica 1
Replica 2
Replica 3
Replication: ASYNCHRONOUS
Failover: MANUAL (promotion)
Replicas serve reads: YES
Data loss: Possible replication lag

🕰️ Point-in-Time Recovery (PITR)

PITR = Daily Snapshot + Continuous Transaction Logs — Restore to any second within 1–35 day window.

Daily Automated Snapshots
AWS takes full DB snapshot every day. Foundation for PITR.
Continuous Transaction Logs
Every INSERT/UPDATE/DELETE logged continuously. Enables second-level precision.
10:05 AM — Developer runs bad DELETE 💥
Accidentally deletes critical production data. Tables emptied. App breaks.
10:30 AM — Incident Detected
CloudWatch alarm fires. On-call engineer checks DB. Data is gone!
PITR → Restore to 10:04:59 AM ✅
RDS Console → Select DB → Actions → Restore to Point in Time → Creates NEW DB instance. Original untouched! RPO: seconds.

🔐 RDS Encryption

At Rest — KMS (AES-256)

App → RDS Engine
       ↓ AES-256 via KMS
[Encrypted EBS Storage]
       ↓ Also encrypted:
Snapshots · Backups · Read Replicas

Transparent to app — no code changes needed.

In Transit — SSL/TLS

Client App
   ↓ SSL/TLS encrypted
RDS Endpoint

AWS provides SSL certificates. Enable via connection string param.

🚨
Critical Constraint: Encryption must be enabled AT DB CREATION TIME. Cannot encrypt existing unencrypted DB!
Workaround: Take snapshot → Copy snapshot WITH encryption → Restore new encrypted DB.

🔑 Data Retrieval Methods

# Example: E-commerce Orders Table
# PK = user_id (Partition Key)  |  SK = order_id (Sort Key)

┌────────────┬──────────┬──────────┬──────────┐
│ user_id(PK)│order_id  │  amount  │   date   │
├────────────┼──────────┼──────────┼──────────┤
│    1001    │ order_1  │  ₹500    │ Jan-01   │
│    1001    │ order_2  │  ₹300    │ Jan-03   │
│    1001    │ order_3  │  ₹750    │ Jan-05   │
│    1002    │ order_1  │  ₹200    │ Jan-02   │
└────────────┴──────────┴──────────┴──────────┘

Method 1: GetItem  → PK=1001, SK=order_1      → O(1) ⚡ Single exact item
Method 2: Query    → PK=1001                   → All orders for user 1001
           + SK:    → SK BETWEEN order_1..order_3 → Range queries
Method 3: Scan     → Read ENTIRE table          → Never in production! Expensive!

# Design Rule: "Access Pattern First"
Bad PK:  country = "India"  → Hot partition (millions in same bucket) ❌
Good PK: user_id            → Evenly distributed ✅

Troubleshooting Scenarios

🔴 EC2 Instance Terminated Unexpectedly
🔑
Key: CloudTrail = WHO · CloudWatch = WHY · ASG = WHAT happened automatically
Step 1: Check state → aws ec2 describe-instances --instance-ids i-xxxxx
Step 2: CloudTrail → Event "TerminateInstances" → WHO / WHEN / SOURCE IP
Step 3: ASG → Health check failed? → Auto-replaced instance?
Step 4: Spot Instance? → AWS terminated for capacity
Step 5: Automation? → Lambda / Terraform / Jenkins cleanup scripts
Recovery:
  EBS snapshot exists? → Create volume → Attach to new EC2
  AMI exists? → Launch new instance
  No backup? → Rebuild from IaC (Terraform apply)
Prevention:
  ✅ Enable termination protection
  ✅ Regular EBS snapshots + AMI backups
  ✅ CloudWatch alarms + SNS notifications
  ✅ IaC (Terraform) for reproducibility
🟡 RDS Storage Full — Production Incident 
# Symptoms: App write failures, "No space left on device"
# CloudWatch: FreeStorageSpace → 0

Immediate Fix (no downtime):
aws rds modify-db-instance \
  --db-instance-identifier mydb \
  --allocated-storage 200 \
  --apply-immediately
# Storage can ONLY INCREASE, never decrease

Root Cause — Find biggest tables:
SELECT table_name, pg_size_pretty(pg_total_relation_size(table_name))
FROM information_schema.tables ORDER BY 2 DESC;

MySQL binary log cleanup:
PURGE BINARY LOGS BEFORE NOW() - INTERVAL 3 DAY;

Prevention:
  ✅ CloudWatch alarm: FreeStorageSpace < 20%
  ✅ Enable Storage Autoscaling in RDS
  ✅ S3 lifecycle for old data → Glacier
  ✅ Log rotation policies
🟣 Developer Deleted S3 / RDS / EC2 — Recovery Plan

Recovery

  • CloudTrail → Who/When/From where
  • EC2 → AMI or EBS snapshot → new instance
  • RDS → PITR to just before deletion
  • S3 → Restore from version (if enabled)

Prevention

  • IAM: deny DeleteObject, TerminateInstances in prod
  • EC2 termination protection
  • RDS deletion protection
  • S3 versioning + MFA delete
  • Terraform/CloudFormation (IaC)
🔵 EC2 Cannot Connect to RDS MySQL 
# Debug Order (Most → Least Common):

1. Security Group (80% of issues!)
   RDS SG inbound: Allow 3306 from EC2-SG  (NOT from IP!)
   Wrong: Allow 3306 from 10.0.1.5
   Right: Allow 3306 from sg-xxxxxxxx

2. Port check:
   nc -zv <RDS-ENDPOINT> 3306

3. Same VPC? Subnet routing?
   EC2 VPC ID = RDS VPC ID?

4. NACL rules:
   Inbound: allow 3306  |  Outbound: allow 1024-65535

5. Credentials:
   mysql -h <RDS-ENDPOINT> -u admin -p

6. DNS:
   nslookup <RDS-ENDPOINT>
   enableDnsSupport + enableDnsHostnames = true
🟠 Docker Container Exits Immediately 
# Diagnose:
docker ps -a
docker logs <container_id>
docker inspect <id> --format='{{.State.ExitCode}}'

# Exit Codes:
0   → Completed normally (process ended as expected)
1   → App crash / unhandled error
137 → OOM killed (SIGKILL) or docker stop
143 → Graceful SIGTERM

# Most Common Fix: Keep process in FOREGROUND!
CMD service nginx start         # ❌ daemonizes, PID 1 exits
CMD ["nginx", "-g", "daemon off;"]  # ✅ foreground, container lives

# Debug: override entrypoint to get a shell
docker run -it --entrypoint /bin/sh myimage

# Golden Rule: Container lives/dies with PID 1. Keep it FOREGROUND!
🟤 ECS Service 502/503 — Memory Leak + ALB Misconfiguration

Root Cause Found

  • Node.js unclosed DB connection pool — memory leak
  • ALB unhealthy_threshold = 10 — routing to sick tasks for 5 min!

Fixes Applied

  • unhealthy_threshold = 3 (was 10)
  • Health interval = 10s (was 30s)
  • /health returns 503 when memory > 80%
  • Fixed pool.end() in app code
🎯 Lesson Learned

"Health checks are your first line of defense. A misconfigured health check routes live traffic to dying services. Make ALB health check tuning a standard part of every deployment checklist."

Interview Cheat Sheet

🛡️ Key Comparisons

  • Multi-AZ = HA (sync) vs Read Replicas = Performance (async)
  • SG = Instance-level, stateful vs NACL = Subnet-level, stateless
  • Snapshot = Manual, exact point vs Backup = Auto + PITR
  • IGW = Bidirectional vs NAT = Outbound only
  • TCP = Reliable+slow vs UDP = Fast+unreliable

⚡ Quick Commands

  • nc -zv RDS-ENDPOINT 3306 — Port check
  • nslookup RDS-ENDPOINT — DNS check
  • docker inspect id --format ExitCode — Exit code
  • systemctl status nginx — Web server
  • aws ec2 describe-instances — EC2 state
  • CloudTrail → TerminateInstances — WHO did it

🎯 One-Line Interview Answers

NAT Gateway: Performs SNAT — translates private IPs to Elastic IP for secure outbound internet access, blocking all unsolicited inbound traffic.

PITR: Combines daily snapshots + continuous transaction logs to restore a DB to any second within the retention window.

SG Referencing: Allow traffic from another SG instead of IP — AWS checks if source ENI has that SG attached, making it dynamic and Auto Scaling-friendly.

Aurora: AWS's proprietary re-engineered DB with decoupled compute+storage, 6 copies across 3 AZs, 5x faster than MySQL.

RDS Proxy: Connection pooling layer that reuses DB connections — critical for Lambda and microservices to prevent connection exhaustion.
